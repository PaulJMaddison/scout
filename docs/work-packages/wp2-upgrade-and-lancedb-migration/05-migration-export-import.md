## Outcome

Scout now has a real local export and validation path behind the public storage adapter boundary. The open-source `scout-postgres` adapter exports current Scout relational records for a tenant, validates the export request, produces deterministic local anchors for Enterprise/Fortress import, and refuses unsupported vector, relationship-set, attribution-path, outcome-event, and private import scopes.

The Enterprise/Fortress import remains a defined contract, not a fake Scout implementation. The private runtime still needs to provide the local LanceDB/vector DB importer, embedding generation, relationship-set construction, retry/dead-letter persistence, and proof against a real local private store.

## Export Format

Contract version:

```text
kynticai.scout.storage-portable-export.v1
```

Implemented API shape:

- `ILocalDataPlaneStorageAdapter.ExportAsync(StorageExportRequest request)`
- Implemented by `ScoutPostgresStorageAdapter` for provider key `scout-postgres`.
- Returns one `StorageExportBatch` per request. Callers continue with `NextCheckpoint` until `IsFinal` is true.

Batch fields:

| Field | Meaning |
| --- | --- |
| `BatchId` | Local export batch ID generated by Scout. |
| `Scope` | Requested `StorageAdapterDataScope`. |
| `Records` | Portable records unless `DryRun` is true. |
| `NextCheckpoint` | Opaque checkpoint for resumable export. |
| `IsFinal` | True when there is no more data for the requested scope. |
| `Errors` | Typed validation errors. |
| `Diagnostics` | Local diagnostics, counts, contract version, and `usesCloudDataPlane=false`. |
| `ValidationReport` | Validation summary and findings for dry run or real export. |

Portable record fields:

| Field | Meaning |
| --- | --- |
| `RecordKind` | Exported kind such as `source_system_event`, `user_signal`, `selector_execution`, `context_fact`, `provenance_metadata`, or `audit_event`. |
| `RecordId` | Stable Scout record GUID. |
| `SourceSystem` | Source-system key where known, or a Scout namespace such as `scout-context`. |
| `SourceRecordId` | Source event ID, source record key, attribute key, entity ID, or stable Scout ID depending on kind. |
| `ObservedAtUtc` | Source observation time or closest local audit time. |
| `Payload` | JSON payload/value/result data preserved from Scout storage. |
| `Provenance` | JSON provenance array or provenance wrapper for the record. |
| `Metadata` | Tenant, audit, source table, correlation, and Enterprise/Fortress anchor metadata. |

Every exported record includes these metadata fields:

```json
{
  "contractVersion": "kynticai.scout.storage-portable-export.v1",
  "adapterKey": "scout-postgres",
  "sourceTable": "source_system_events",
  "tenantContext": {
    "tenantId": "<scout tenant guid>",
    "tenantSlug": "<tenant slug>",
    "layer": "<tenant slug>"
  },
  "fortressAnchor": {
    "entity_type": "source_system_event",
    "postgres_pk": "source_system_events:<scout record guid>",
    "layer": "<tenant slug>"
  },
  "createdAtUtc": "<timestamp>",
  "updatedAtUtc": "<timestamp>",
  "usesCloudDataPlane": false
}
```

Implemented export scopes:

| Scope | Record kind | Preserved data |
| --- | --- | --- |
| `SourceEvents` | `source_system_event` | Event ID, source system, event type, payload JSON, headers JSON, external user/account IDs, workspace/data-source/user IDs, status, correlation ID, processing/dead-letter metadata, timestamps, and source-event provenance wrapper. |
| `UserSignals` | `user_signal` | Signal key, value JSON, value type, user/data-source IDs, observed time, provenance JSON, audit timestamps, tenant/layer metadata. |
| `SelectorExecutions` | `selector_execution` | Selector/user IDs, correlation ID, status, execution mode, result value/type/confidence, raw source data, validation errors, pipeline trace, result provenance, requested/started/completed timestamps, and errors. |
| `ContextFacts` | `context_fact` | Attribute key, value JSON, value type, confidence, explanation, snapshot/semantic-selector IDs, observed/freshness timestamps, provenance JSON. |
| `Provenance` | `provenance_metadata` | Kind, source system, source record key, metadata JSON, selector/context-fact links, observed and audit timestamps. |
| `AuditEvents` | `audit_event` | Actor, action, entity type/id, correlation ID, metadata JSON, before/after JSON, and created timestamp. |

Unsupported in Scout/open-core export:

- `DataItems`
- `RelationshipSets`
- `AttributionPaths`
- `OutcomeEvents`
- `Vectors`

Those scopes require private/local Enterprise/Fortress contracts or future public Scout persistence. The exporter reports them as validation errors rather than silently emitting incomplete data.

## Import Contract

Enterprise/Fortress must provide the local importer. Scout/open-core does not fake it.

Required private import entrypoint:

```text
ILocalDataPlaneStorageAdapter.ImportAsync(StorageImportRequest request)
```

or an equivalent local-only Enterprise/Fortress API/script that consumes `kynticai.scout.storage-portable-export.v1` batches and reports equivalent result fields.

Required import behaviour:

- Run only inside the customer-owned environment.
- Reject records whose `contractVersion` is unsupported.
- Reject records where `tenantContext.tenantId`, `tenantContext.tenantSlug`, request tenant, and `fortressAnchor.layer` do not match.
- Treat `fortressAnchor.postgres_pk`, `fortressAnchor.entity_type`, `fortressAnchor.layer`, and request idempotency key as the upsert/idempotency boundary.
- Preserve source event IDs, selector IDs, source record IDs, provenance references, correlation IDs, and audit timestamps.
- Preserve relationship inputs where they exist in Scout exports, but do not infer canonical relationship sets from incomplete Scout fallback records without marking the import partial.
- Generate embeddings locally if vectors are required. LanceDB writes require dense local embeddings and must skip, dead-letter, or fail null embeddings; they must not be faked.
- Write LanceDB/vector DB rows through the private local vector contract using `id`, `entity_type`, `postgres_pk`, `layer`, `embedding`, `metadata_json`, `created_at`, and `updated_at`.
- Write relationship sets, attribution paths, outcome events, checkpoints, dead letters, and retry metadata to private local stores only.
- Return a validation report and import result with imported, skipped, failed, retryable, and dead-letter counts.
- Never use Cloud as staging, backup, search, import, vector, dead-letter, or support-bundle storage for customer data.

## Dry Run Behaviour

`StorageExportRequest.DryRun=true`:

- Resolves and validates tenant context.
- Checks requested scopes and unsupported scopes.
- Reads matching local Scout records for count and JSON validation.
- Validates checkpoint shape if supplied.
- Produces `ValidationReport`.
- Returns no portable records.
- Sets `Diagnostics.usesCloudDataPlane=false`.
- Does not write Scout tables.
- Does not call Enterprise/Fortress.
- Does not call Cloud.

`StorageImportRequest.DryRun=true` is now present in the public contract, but Scout/open-core still returns `NotSupported` for import. The private Enterprise/Fortress importer must use dry run for local preflight validation before writing private tables or vector stores.

## Validation Checks

Implemented in Scout export:

- Tenant ID exists in local Scout storage.
- Tenant ID and slug match; mismatches fail closed to avoid cross-tenant leakage.
- Scope is not empty.
- `MaxRecords` is greater than zero.
- Unsupported Scout scopes are reported as validation errors.
- Checkpoints are opaque base64 JSON and must match a record in the current export set.
- JSON object fields parse as objects where object shape is required.
- JSON array fields parse as arrays where provenance array shape is required.
- Counts by record kind are reported.
- Every batch reports `usesCloudDataPlane=false`.

Required in Enterprise/Fortress import:

- Contract version compatibility.
- Tenant/layer consistency across request, record metadata, vector writes, relationship traversal, and governed JSON handoff.
- Idempotent upsert behaviour.
- Dense embedding availability for LanceDB/vector writes.
- Citation/provenance coverage before relationship-set or attribution-path indexing.
- Private retry/dead-letter status and resumable checkpoints.
- No Cloud data-plane access.

## Failure Modes

Scout exporter failure modes:

- Local Scout database unavailable: health check reports `Unavailable`; export fails through EF query errors.
- Tenant not found: validation error, no records returned.
- Tenant ID/slug mismatch: validation error, no records returned.
- Unsupported scope such as `Vectors` or `RelationshipSets`: validation error; supported scopes can still be exported if requested in the same call.
- Invalid checkpoint: validation error, no records returned.
- Invalid JSON in a Scout JSON field: validation finding and typed error.
- Requested scope has no records: valid empty export if tenant and scope are otherwise valid.

Enterprise/Fortress importer failure modes to handle locally:

- Contract version unsupported.
- Missing private adapter/runtime.
- LanceDB/native dependency unavailable.
- Local embedding model/tokenizer unavailable.
- Null or wrong-dimension embedding for a LanceDB row.
- pgvector extension/table unavailable if pgvector fallback is configured.
- Relationship input missing provenance or citation coverage.
- Cross-tenant/layer mismatch.
- Partial import after retryable store failure.
- Dead-letter queue/checkpoint write failure.
- Operator disk space or backup preflight failure.

## Rollback Guidance

Before migration:

- Take a Scout database backup.
- Record Scout version, config, Docker compose state, connector package versions, and storage adapter settings.
- If existing Enterprise/Fortress private stores exist, snapshot their local volumes before import.
- Run Scout dry-run export and private import dry run before any routing switch.

If Scout export fails:

- Leave Scout unchanged.
- Fix local database, tenant, scope, checkpoint, or JSON issues.
- Re-run dry run before real export.

If Enterprise/Fortress import fails before routing switches:

- Keep Scout as the active data plane.
- Disable the Enterprise/Fortress provider in local config.
- Keep private import logs, checkpoints, and dead letters local for operator review.
- Re-run import from the last successful local checkpoint after fixing the cause.

If import fails after routing switches:

- Switch local provider config back to the Scout-compatible path.
- Stop private Enterprise/Fortress runtime services.
- Rebuild private vector/relationship stores from Scout export where possible.
- Restore Scout database only if public Scout migrations changed data and cannot be safely rolled forward.

If Cloud licence or entitlement changes:

- Cloud may disable future private downloads or updates.
- Cloud licence rollback must not delete Scout, Enterprise/Fortress, LanceDB, vector, relationship, checkpoint, or dead-letter data.
- Deleting private local stores must require explicit operator action.

## Current Blockers

- The private Enterprise/Fortress import contract is not implemented in Scout/open-core and should not be faked.
- Scout does not persist canonical relationship sets, attribution paths, outcome events, vectors, or exact data items as first-class public tables.
- LanceDB/native-store, pgvector, and local embedding model proof remain private/local opt-in validation work.
- A polished operator CLI wrapper is specified but not implemented.
- xhigh review gates are still required before this data-model/storage/security-sensitive migration work is marked complete.
